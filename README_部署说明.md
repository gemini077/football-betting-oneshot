# 足球融合模型｜ChatGPT Work 部署包

- 模型名称：`Football Betting OneShot`
- 内部标识：`football-betting-oneshot`
- 当前版本：`v0.12.0`
- 版本规则：普通修复升补丁版本，新模块或正式规则升次版本，架构级不兼容大改升主版本。

统一赛事工作台入口：`data/match_workspace/latest.html`。它把每日体彩赛程、用户选择的待分析场次、赛前报告和赛后复盘放在同一页面；每日更新只刷新赛程，不会自动分析全部比赛。

`v0.12.0` 将严格复盘口径扩展到全部玩法：唯一主维度按玩法、选项、盘口线和结算时段精确匹配；亚洲盘区分全赢、半赢、走水、半输、全输；并列玩法、备选、Top-N和相关市场不得替代唯一主维度命中。

`v0.11.1` 收紧比分复盘口径：只有唯一首推与实际90分钟比分完全一致才算命中；相邻比分只作未命中误差诊断，Top-N不得扩展为命中覆盖。

`v0.11.0` 重构赛程与复盘联动：已复盘比赛自动移入已完赛区，历史业务日不能覆盖固定主页；单场复盘覆盖全部玩法维度、时间轴、错点、反事实和模型修正；赛前分析、单场赛后复盘与复盘总表成为三个明确入口，并保留当前比赛上下文。

## 推荐架构

- **ChatGPT Project**：保存项目指令、长期对话与少量核心知识文件。
- **桌面端 Work + 本地文件夹**：作为日常赛前分析、截图/JSON读取、报告生成和工作簿维护的主工作区。
- **Codex（可选）**：仅负责抓取脚本、API适配、自动化和代码维护；不负责替代足球模型结论。

这套结构比“只创建一个自定义 GPT”更适合长期模型，因为模型需要：
1. 固定的分析规程；
2. 会持续增长的赛前/赛后样本；
3. 可修改的运行状态与资金台账；
4. 可回滚的模型修正规则；
5. 本地 Excel 工作簿。

## Windows 部署步骤

1. 将本压缩包解压到固定目录，例如：
   `D:\FootballModelWork\`
2. 打开 ChatGPT 桌面端。
3. 新建或打开一个 Project，命名为：
   `足球融合模型`
4. 把 `00_PROJECT_INSTRUCTIONS_粘贴到项目指令.md` 的全文粘贴到项目指令。
5. 在桌面端模式切换器中选择 **Work**。
6. 选择“打开本地文件夹”，打开解压后的目录。
7. 首次对话粘贴下面的启动命令。

## 每日工作方式

### 赛前
只需给出比赛、日期或赛事。系统先按 `09_DATA_FETCHING_数据抓取.md` 自动抓取公开赛程和赔率；截图、私有JSON与实际购买价可随后补充。然后说：

```text
按完整融合模型分析这些比赛。先做数据完整性审计，再逐场输出完整报告，最后单独输出投注层。当前节点是T-___。
```

本地抓取入口：

```powershell
python scripts\fetch_football_data.py --date YYYY-MM-DD --match 球队名 --deep
```

抓取完成后可生成数据审计网页：

```powershell
python scripts\generate_analysis_report.py --fetch-manifest data\fetch_runs\YYYYMMDD_HHMMSS\YYYYMMDD_HHMMSS_fetch_manifest.json
```

完整融合分析后，再通过 `--analysis-json` 叠加模型结果，生成完整分析网页。详细字段、固定版块和锁单约束见 `10_HTML_REPORT_网页报告.md`；输出统一存入 `data/analysis_reports/YYYYMMDD_HHMMSS/`。

### 锁单
只有你明确输入：

```text
锁单：比赛｜玩法｜赔率｜金额
```

系统才可以更新运行状态和工作簿。

### 赛后
输入：

```text
比赛已结束。核验90分钟赛果，按9层复盘，更新模型修正池、运行状态和Excel工作簿。
```

## 重要说明

- `05_RUNTIME_STATE.json` 是当前状态的唯一真源。
- 历史复盘工作簿属于可选外部状态文件。将最新版 `.xlsx` 放入 `data/postmatch_reviews/` 后，再在 `05_RUNTIME_STATE.json` 中填写相对路径。
- 所有赛前预测按90分钟结算；加时与点球单独记录。
- 赔率方向与投注价值必须分开判断。
- 不做走地、不追单、不因为“当天场次少”降低执行线。
- 网页报告只是分析交付形式；生成报告不会改变余额或注单，只有明确“锁单/已下单”才更新运行状态。

## 实时/渠道价EV复算（v0.5.0）

`v0.5.0` 新增同一盘口合约的价格联动层。它按比赛、玩法、盘口线和选项精确匹配桥接最新价，也支持报告后输入用户渠道十进制赔率；通过开放状态、赔率尺度和新鲜度检查后，复算点估计EV、保守EV与最低可接受赔率。盘口线变化时强制重算事件概率，不会把新价格套到旧概率上。

```powershell
python scripts\live_ev_reprice.py --request examples\live_ev_reprice_request.json
```

结果写入 `data/live_ev_reprices/YYYYMMDD_HHMMSS/`。该模块只输出“候选/空仓/影子验证”，固定不授权执行、不修改余额、不写入锁单。

## 网页实时EV悬浮窗（v0.8.0）

`v0.6.0` 把EV复算器嵌入目标比赛详情页；`v0.7.0` 增加分析报告自动赋值；`v0.7.1` 合并同名玩法并按“玩法类别→盘口线/比分→方向”选择，补入波胆；`v0.7.11` 将全场普通/高倍波胆合并为“全场波胆”，并将上半场普通/高倍波胆合并为“上半场波胆”；`v0.7.12` 将波胆改为比分按钮矩阵；`v0.7.13` 将全部选择统一改为矩阵；`v0.7.14` 根据实际使用反馈恢复“玩法类别”下拉框，具体盘口线、方向、比分和实时赔率继续使用矩阵按钮。底层仍保留原始玩法代码和盘口ID，报告自动绑定时继续精确匹配原始合约。悬浮窗每秒从本地服务读取同一盘口的最新已核验价格，实时显示当前赔率、点估计EV、保守EV、最低可接受赔率、报价年龄、Kelly诊断和固定小额候选金额。报告生成后按比赛ID自动同步合约、概率和EV门槛，不再需要手工抄写。

首次升级后在 `chrome://extensions/` 点击一次“重新加载”，再刷新比赛详情页。扩展弹窗可随时显示或隐藏悬浮窗。

悬浮窗只接受报告生成器发布并确认的模型概率；没有当前比赛配置、报告为空仓或字段不完整时会清空候选金额。整数盘和四分之一亚洲盘不会套用简单EV公式；比赛进入赛中状态后，赛前概率自动失效并禁止给出金额。悬浮窗没有下注按钮，不读取页面DOM或网站账户余额，不提交订单，也不改变锁单状态。

## Polymarket公开市场证据（v0.8.0）

`v0.8.0` 新增 Polymarket 足球市场的公开只读分析源。分析比赛时可抓取对应赛事的90分钟胜平负、大小球、双方进球、波胆等公开合约，以及买一、卖一、中间价、流动性、成交量、费用配置和结算规则，用于检查市场共识、分歧、拥挤度与价格变化。

## 自动赛后复盘与网页台账（v0.9.0）

`v0.9.0` 新增赛后自动队列、双源90分钟赛果核验、固定9层复盘、按记录ID递增更新Excel，以及自包含网页台账。固定网页入口为 `data/postmatch_dashboard/latest.html`。自动流程只处理已有完整赛前概率报告，未明确锁单的候选不会写成已投注。

`v0.9.1` 将赛后复盘改为按比赛开赛时间的一次性唤醒：普通比赛开赛后135分钟，单回合淘汰赛开赛后195分钟；未完赛仅45分钟后重试一次，取消固定30分钟轮询以避免空跑token。

该模块不登录、不连接钱包或账户、不读取持仓，也没有下单或撤单能力。Polymarket价格不是用户的可成交渠道价；未完成历史时间外校准前，它只显示在报告的“市场证据层”，不直接修改模型概率，不参与EV、Kelly或投注金额。最终EV仍在用户提供实际渠道赔率后复算。

中文队名无法可靠绑定时，抓取命令可显式提供英文别名：

```powershell
python scripts\fetch_football_data.py --date 2026-07-14 --match 法国 --polymarket-home France --polymarket-away Spain --polymarket-kickoff 2026-07-14T19:00:00Z
```

完整边界和质量门禁见 `12_POLYMARKET_PUBLIC_ANALYSIS_只读市场证据.md`。
