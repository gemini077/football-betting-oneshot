# 输出模板

## 赛前报告

```text
# [主队] vs [客队]｜T-___完整分析

0. 报告类型与时间窗
1. 数据加载与队名校验
2. 基本面前置
3. news / form-h2h / standings
4. 战意—阵容—盘口自洽审计
5. 盘口公司分类与时间轴
6. 欧亚理论盘型校验
7. 盘型开大/开小/适中审计
8. 多市场拆解
9. 正确比分三组拆
10. 去水与概率修正
11. 初盘—临盘诱盘/阻盘
12. 经验库触发
13. 正诱/反诱/挡盘
14. 三维交叉验证
15. 6D/MBI/平局/偏离
16. 凯利/交易所/机构盘赔
17. 竞彩买量与赔付暴露
18. Poisson+DC矩阵
19. 总进球桶
20. BTTS
21. 胜平负/让球
22. 唯一主维度
23. 唯一首推比分
24. 最大错点
25. betting-SOP状态
```

在第15步前后必须插入“基础内核审计面板”，至少包含：

```text
30家公司分层共识：已计算/降级/未计算
SCS：数值、方向、量价验证状态
DRI：原始值、校准值、风险层级
Lead-Lag：领涨公司、跟随链、时间质量
水位流向：同盘口独立源数量、流入/流出比
交易所背离：方向、成交量质量、是否降权
页面凯利指数共识：三层均值、冲突方向（非仓位Kelly公式）
Shin去水：三向概率、过圆率、理论返还率
6D：D1—D5连续分数、权重、0—6旧制总分、限制与修正动作（知识库的D6已合并）
陷阱：上游声称49条、当前明确定义47条；显示已评估数、不可评估原因、触发编号和A/B/C分类
机器学习审计：模型版本、训练/验证/校准/测试时间、特征as-of、泄漏检查、校准方法、Brier/log loss、基线增量
回测口径：持有期收益、年化收益、自然日、ROI、CLV、最大回撤、Sharpe频率与费用
基础内核完成状态：完整/降级/不完整
```

基础内核不是“已计算”时，报告标题不得写“完整分析版”。

大小球模块至少列出：

```text
模型期望总进球：
xG供应商/模型版本/是否含点球/数据截止时间：
主客场及对手校正后的xG攻防强度：
赛前λ生成方法与联赛基准：
理论大小球线（公平大球价最接近2.00）：
理论线大球公平十进制赔率：
实际开盘/即时盘口与两侧十进制赔率：
实际盘口大球/小球公平赔率：
0、1、2、3、4、5、6+球概率：
离散总进球组合概率/公平赔率（如市场提供1或3球、3或4球）：
阵容/战术/天气/赛程修正及证据：
模型与市场是否冲突：
结论：候选/观察/空仓
```

不得只凭“升盘、退盘、降水”输出诱大/诱小结论；换线后必须展示新盘口的完整亚洲结算与价格审核。

若分析半全场市场，模块至少列出：

```text
标签口径：固定主队视角
数据源/赛事/赛季/起止日期：
总样本量/当前筛选样本量/强队定义：
上半场主客λ：
下半场主客λ：
分时段进球风险率及暴露分钟（含比分状态/红牌/主客/对手校正）：
H/H、H/D、H/A、D/H、D/D、D/A、A/H、A/D、A/A联合概率：
9项概率合计/矩阵遗漏质量：
半场比分状态条件化状态：已校准/未校准/降级
实际9项十进制赔率及时间：
半全场9项过圆率/理论返还率：
同公司同时间1X2过圆率/理论返还率：
可成交限额/报价新鲜度/拒单与流动性风险：
各项公平赔率/点估计EV/保守EV：
换人、红牌、阵型和天气修正证据：
八类证据审计（状态/上半场/下半场转换/伤停轮换/情境动机/价格/教练/心理代理）：
每类数据源/样本量/对手校正/赛前截止时间/缺失状态：
共线与重复计权检查：已完成/未完成
八类特征到9项联合概率的校准映射：已校准/未校准/降级
结论：候选/观察/空仓
```

不得将按热门方描述的“平胜、负胜”直接填入主队视角合约；没有9项概率、实际价格和口径映射时只能输出数据缺失或观察。

完成文本分析后，默认同时生成HTML可视化报告，结构和状态约束见 `10_HTML_REPORT_网页报告.md`。仅有抓取快照、尚未形成完整模型结论时，HTML标题和状态必须明确写“数据审计版”，不得输出伪造的唯一比分、唯一主维度或投注建议。

若存在多张票，必须额外输出：

```text
统一主页三层组合视图：保本层 / 中轴层 / 博上层各自候选数、候选暴露、已锁暴露
统一主页跨场串联表：票号、组成腿、各腿独立价格审核、组合赔率、联合概率、组合EV、相关性、金额、取消条件
统一主页同场重复暴露审计：比赛、涉及票号、共同失败情景、总暴露、是否受控

单场分析页不显示三层组合和跨场串联，只显示本场候选合约与价格审核。
```

三层可以全部为空；不得为了页面完整而硬填候选或凑串关。

HTML投注层沿用相同字段。只有用户明确说“锁单/已下单”后，状态才可显示“已锁单”；其余只能显示“候选”或“空仓”。

若本场使用实时EV悬浮窗，投注层候选还必须带 `live_ev_profile`：精确比赛ID、玩法代码/名称、盘口线、选项代码/名称、结算类型、点估计概率、保守概率、概率来源与校准状态、报价时效和保守EV执行线。报告生成后自动发布到本地悬浮窗。分析结论为空仓、字段不完整或多候选未指定唯一 `overlay_primary` 时，必须发布非活动配置并把悬浮窗候选金额清零；不得沿用旧值。
