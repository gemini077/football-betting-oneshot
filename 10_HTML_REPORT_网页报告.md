# HTML可视化报告

完整模型分析后，默认生成一个自包含、可直接打开和分享的HTML报告。视觉采用暗色数据仪表盘，但内容严格遵守本项目的唯一比分、唯一主维度、最大错点、价格纪律和锁单规则。

## 生成命令

仅有抓取数据时，生成“数据审计版”：

```powershell
python scripts\generate_analysis_report.py --fetch-manifest data\fetch_runs\YYYYMMDD_HHMMSS\YYYYMMDD_HHMMSS_fetch_manifest.json
```

完整分析后，叠加模型结果：

```powershell
python scripts\generate_analysis_report.py `
  --fetch-manifest data\fetch_runs\YYYYMMDD_HHMMSS\YYYYMMDD_HHMMSS_fetch_manifest.json `
  --analysis-json data\analysis_inputs\YYYYMMDD_HHMMSS_分析结果.json
```

分析结果字段遵循 `schemas/analysis_report.schema.json`。

若分析JSON含 `match.live_match_id` 和完整的候选 `live_ev_profile`，生成报告时会同时发布 `data/live_ev_profiles/current/<match_id>.json`，悬浮窗按比赛ID自动填入合约、点估计概率、保守概率、EV执行线和报价时效。没有合格候选时会发布非活动配置以清除旧值；这一步只同步分析输入，不代表锁单。

## 固定页面结构

页面采用“结论优先、推导随后”的阅读顺序：

1. 比赛、时间窗和快照信息；
2. 核心答案：唯一主维度、唯一比分、投注状态、价格结论、首要错点；
3. 模型胜平负概率、预期进球、主维度点概率、保守概率边界、公平赔率和最低执行赔率；
4. 模型概率与市场去水概率的逐项差值；
5. 盘口、赔率和让球市场解读；只有至少3个真实独立快照时才显示赔率轨迹；
6. 基本面与比赛剧本；
7. 总进球、BTTS及λ变化后的参数敏感性；
8. 0—5球比分热力矩阵、比分排序和唯一首推比分；
9. EV价格审核、实时渠道复算与最终投注判断；
10. 结论依据链；
11. 失效条件与重算触发点。

概率区间采用“保守概率边界—模型点估计”的执行表达，不标成统计置信区间。参数敏感性只是压力测试，不覆盖正式概率。模型与市场分歧只用于定位需要解释的方向，不自动等于正EV。

若报告含活动的 `live_ev_profile`，页面每3秒向本机 `http://127.0.0.1:8765/v1/reprice` 请求同一合约的新鲜渠道价，并显示实时赔率、保守EV、执行线和结论。匹配键必须同时包含比赛、玩法、盘口线和选项；盘口换线必须重算事件概率。比赛进入非赛前状态后，页面停止沿用赛前概率计算滚球EV。

数据完整性、基础内核、MBI、6D、规则扫描和原始市场明细保留在页面底部的“模型内部计算记录”中，默认收起，不占用主报告阅读层级。

单场报告只展示本场的分析结论、价格审核、实时EV和候选合约，不展示保本层、中轴层、博上层，也不在本页拼接其他比赛。三层组合、跨场串联和同场重复暴露审计统一由 `data/match_workspace/latest.html` 汇总展示；没有合格价格时三层均可空仓，不得凑数。

## 状态纪律

- 数据不完整时只能生成“数据审计版”，不得伪造模型概率、唯一比分或投注价值。
- 完整分析结果写入后才能生成“完整分析版”。
- `final_execution_version=true` 只表示数据已冻结为最终执行窗口，不代表已锁单。
- 未经用户明确说“锁单/已下单”，页面中的方案只能标记为“候选”或“空仓”。
- 首份完整报告的投注层使用最新抓取赔率计算初算EV，并展示赔率来源和时间；用户反馈实际渠道盘口与赔率后，重新生成报告或投注层，展示渠道价复算EV。
- 用户渠道价复算结果只用于决定候选、取消或空仓，不自动改变锁单状态。
- 实时页面即使显示“价格通过”，也只能标为候选；未经用户明确说“锁单/已下单”，不得改变余额、暴露或未结算注单。
- 报告存放于 `data/analysis_reports/YYYYMMDD_HHMMSS/`，文件名使用日期时间，不使用递增版本号。
